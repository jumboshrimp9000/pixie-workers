# ProfitPath Bulk Fulfillment Learnings

This is a running ops note from the one-off ProfitPath replacement run.

## Immediate Process Changes

- Separate DNS preparation from provider provisioning. Create/get Cloudflare zones and move registrar nameservers for all domains first, then enqueue Microsoft, Azure, Google, or SMTP provisioning only for domains that are active in Cloudflare.
- Treat registrar API success and registry delegation as different states. Porkbun can show Cloudflare nameservers before public/authoritative delegation has aged out.
- Do not make a whole batch wait on the slowest DNS domain. Track per-domain readiness and seed/provision domains as they become ready.
- Use retry wrappers around Cloudflare and Porkbun reads/writes. Both APIs returned transient 5xx responses during bulk operation.
- Launch Microsoft workers with a longer action lease for 25+ inbox domains. A 99-inbox Azure domain can exceed the default stale-action window.
- For one-off migrations, seed customer/workspace/domain/inbox rows first so the UI can reflect true operational state while workers run.
- Store sending-tool settings at the domain/order level before upload. The upload worker should prefer order-specific settings over credential defaults so bulk jobs do not inherit stale workspace defaults.

## Worker Throughput Observations

- Azure shared mailbox creation is currently sequential per domain.
- Observed mailbox creation pace is roughly 4-8 seconds per mailbox after Exchange sync.
- Eight parallel worker loops on different admins/tenants are working without obvious domain collisions, but the 4 GB Microsoft worker needs swap or fewer loops to avoid memory pressure.
- Normal Simple Inboxes Azure orders are 25 inboxes, so one domain should usually be much shorter than this 99-inbox stress case.
- Instantly Microsoft OAuth upload was a larger bottleneck than mailbox creation when run one mailbox at a time. The uploader now needs configurable per-domain OAuth concurrency with isolated Playwright contexts.
- The API upload worker must have action heartbeats/long leases too. A 99-inbox OAuth upload can run long enough for an old worker to reclaim the same action and create duplicate upload attempts.
- If provider provisioning creates a separate upload action, the original provider action needs a real waiting state or heartbeat while upload runs. Parking it as `in_progress` without a heartbeat can make it stale and reclaimable before a large upload finishes.
- Do not requeue provider actions every couple minutes while upload is pending. Bulk workers will keep reacquiring completed provider domains, reconnecting to Microsoft, and wasting slots. Park the provider action with a long safety retry, roughly 30 minutes for 99-inbox uploads, while the upload action finalizes it directly.
- In same-day high-volume runs, the provider upload-wait retry may need to be longer than 30 minutes or dynamically extended while the upload action is still pending/in progress. Otherwise those already-provisioned domains re-enter the top of the Microsoft queue and delay fresh DNS-ready work.
- Apply `next_retry_at` eligibility in the Supabase query before applying the limit. Fetching the oldest pending actions first and then filtering future retry times in PowerShell can let future-dated upload waits hide fresh DNS-ready domains behind them.
- During a live run, long-lived PowerShell workers keep old queue code in memory. If they cannot be safely restarted, use a narrow temporary queue-keeper that only moves `Sending-tool upload pending` provider actions out of `pending`; otherwise fresh DNS-ready domains can be hidden behind completed-provider/upload-wait records.
- When parking an action with no retry time, send JSON `null` for `next_retry_at`, never an empty string. PowerShell string parameters can turn `$null` into `""`, which Supabase rejects for timestamp fields and leaves noisy partial updates in the worker logs.
- Instantly's OAuth success endpoint can be ahead of the account-read endpoint. Treat confirmed OAuth success as authoritative, then retry account read/settings PATCH until the account becomes visible.
- Instantly account-read and OAuth-init endpoints rate-limit before the documented workspace-wide ceiling in this workflow. Use a low account-validation concurrency, retry 429s with backoff, and tune Microsoft OAuth to a steady rate rather than maxing browser contexts.
- Before re-running a failed/interrupted upload action, check which accounts already exist in Instantly and skip OAuth for those accounts. This makes upload retries resumable instead of redoing every OAuth login.
- For one-domain upload batches, do not check existing Instantly accounts one email at a time. Use the accounts list/search endpoint for the domain, filter the returned emails locally, and only fall back to direct per-email reads if the domain search cannot complete. This turned a 99-call preflight into roughly one call per domain and reduced 429 pressure.
- Instantly reply-rate reads must be normalized as a human percent contract. In this workflow, `0.6` and `60` both mean `60%`; if the verifier compares raw API values to `60`, it can falsely classify healthy accounts as bad and fan out into thousands of unnecessary direct account reads.
- Instantly can reject a new OAuth upload with "account already exists in another workspace" even when none of our stored workspace API keys can read that account. For a same-day bulk migration, the practical recovery is to retire that persona from the active Simple Inboxes set, create a replacement mailbox/persona on the same Microsoft admin, and retry upload with domain-level preflight so only the missing replacement account is OAuth-connected.
- A replacement-mailbox repair must pause the existing upload action while the new Microsoft mailbox is being created. If the upload worker runs while the domain has only 98 active target rows, it can incorrectly complete a weak 98/99 result or churn on the old blocked address.
- Running the Microsoft PowerShell mailbox pipeline outside the normal `run.ps1` claim path can lose the action lease fence and skip final action updates. Use the normal claim/orchestrator path or be prepared to reconcile the provider action manually after the provider-side proof succeeds.
- Before manually requeueing a domain, re-read the action row and confirm it is not already `in_progress`. During `acquisitionadvantage.info`, a manual requeue raced a live worker that had started creating mailboxes successfully, and the action state had to be repaired back to `in_progress` so finalization could succeed.
- Recovery reruns that find no pending inbox rows may use all active inbox rows as the working set. That is safe but expensive for a 99-inbox domain, and it can reset every mailbox password before upload. Prefer a dedicated single-mailbox recovery path for production so the worker does not re-touch the whole tenant just to replace one blocked persona.
- When changing upload validation behavior during a live migration, restart the standalone upload workers. Long-running Node workers keep the old verifier in memory, so a patched `dist` file alone will not stop repeated false failures.
- Keep the number of simultaneous Instantly domains lower than the Microsoft provisioning lane count. Mailbox creation can run wider across tenants; Instantly account reads/settings validation are shared against one workspace API key and need a steadier domain width.
- When launching ad-hoc worker lanes, use exact process-argument checks. Broad `pgrep` patterns can match the launcher shell itself and falsely report that a lane is already running.
- Upload width can be nudged upward after a long clean run, but watch host load and swap, not just free memory. Moving the API host from 14 to 15 active upload domains was possible, but it immediately increased load and swap, so that should be treated as the current ceiling for this box.
- If one mailbox fails transiently after the domain has advanced to SMTP/DKIM states, the retry must rewind to mailbox creation when any pending inbox rows remain. Otherwise the worker skips the missing inbox and only repeats late-stage validation.
- Exchange aggregate mailbox counts can lag individual `New-Mailbox` success responses. Do not upload while Exchange count is lower than expected, but treat that state as a timed retry with a clear log rather than an immediate hard failure.
- A DKIM-complete retry with no pending inbox rows still needs to reconnect to Exchange and re-count mailboxes before queuing upload. Passing `actual=0` into finalization skips the exact validation the retry was meant to enforce.

## Simple Inboxes Worker Recommendations

- Add an explicit `dns_preflight` queue/state:
  - create Cloudflare zone
  - update registrar nameservers
  - poll authoritative delegation
  - poll Cloudflare active
  - only then enqueue provider provisioning
- Use the same Cloudflare-active readiness gate for Azure, Microsoft 365, Google, and SMTP-backed orders. The provider-specific worker should not start until DNS ownership is confirmed.
- Add per-admin concurrency locks, but allow global parallelism across different Microsoft/Google admins.
- Make worker lease duration proportional to expected inbox count.
- Persist progress at mailbox granularity so a retry can resume from the next missing mailbox instead of treating the domain as a monolithic step.
- Do not run SMTP AUTH / IMAP enablement as a per-mailbox loop. Treat it as domain/tenant-level readiness, then validate once before upload.
- DKIM completion must require tenant-side enablement. Adding selector CNAMEs to Cloudflare is not enough; `Set-DkimSigningConfig -Enabled $true` must succeed and `Get-DkimSigningConfig` must report `Enabled=true` before upload.
- Linux PowerShell images do not have `Resolve-DnsName`, `dig`, or `nslookup` by default in our current Microsoft container. DKIM DNS checks need a DNS-over-HTTPS fallback, or the worker will waste time and report false negatives even when Cloudflare is already serving the CNAMEs.
- Suppress Microsoft cmdlet confirmation prompts globally in non-interactive bulk runs. Even prompts that default to Yes add noisy log output and can become a hidden blocking risk under different shells/modules.
- Review the Microsoft email-service enable order before making Azure bulk live. Some tenants returned harmless Graph 400s when `supportedServices += Email` was attempted before all service records were published; the worker recovered later, but this adds avoidable seconds and log noise per domain.
- Watch PowerShell helper functions for accidental pipeline output. `Add-DomainToM365` leaked the Graph POST response before returning its status hashtable, and a multi-item return value made `$addResult.IsVerified` truthy even when both values were false. This caused premature "Domain Verified" flow and the email-service 400 churn.
- After suppressing the leaked Graph POST output, new Microsoft domains followed the intended order: add domain, publish verification TXT, verify, enable Email service, then publish service DNS. This removes the repeated pre-verification 400s and should be the scale path.
- Do not keep a Microsoft worker occupied for 15+ minutes waiting on tenant DKIM acceptance. After the CNAME is visible and a short enable retry fails, defer the action and let the worker create the next domain.
- Once DKIM CNAME visibility is proven, keep the deferred retry short. Fifteen minutes was too conservative for a one-off migration; five-minute retries are a better bulk default while Microsoft catches up.
- Use provider-side validation as the completion gate:
  - Microsoft domain verified
  - Exchange accepted domain present
  - Google Workspace domain verified, if provider is Google
  - SMTP provider DNS requirements present, if provider is SMTP
  - expected mailbox count exists
  - DKIM selectors present/enabled
  - Instantly upload action completed and confirmed expected inbox count
- Start Instantly upload workers independently from Microsoft workers so upload does not block mailbox creation capacity.
- Make mailbox persona generation explicit. For this replacement run we reused the old Airtable `First Last Name JSON` / `Created User JSON` personas for 1:1 continuity; for normal Simple Inboxes Azure orders, default to the approved common U.S. female-name pool and store the selected persona list on the order/domain so retries are deterministic.
- Add an upload worker pool separate from the API web process. It should support per-tool concurrency controls, e.g. Microsoft OAuth contexts per domain and number of domains uploading in parallel.
- For Instantly OAuth, expose two different throttles: account-read validation concurrency and browser OAuth concurrency. Validation should usually be much lower because it is easy to accidentally create a burst across many domains.
- Add a third Instantly throttle for number of domains uploading at once. Six total upload lanes with low account-validation concurrency behaved more safely than thirteen lanes, even when each lane only used two OAuth browser contexts.
- Scale Instantly upload by adding more domain workers at low per-domain OAuth concurrency before increasing per-domain browser concurrency. This gives better control and avoids the bursty behavior that triggered 429s.
- For retries after partial upload, use domain-level Instantly account search before OAuth. The worker should log `existing/target` counts and then upload only the missing tail; this is the critical path for resumability after an interrupted browser/OAuth run.
- Any server running multiple Playwright/OAuth workers needs swap or explicit memory guardrails. The API box also needed swap once upload lanes were increased beyond the web worker plus four upload workers.
- Microsoft PowerShell workers also need memory guardrails. The original 4 GB Microsoft box filled its 4 GB swap under seven provisioning lanes, so adding temporary swap was safer than killing active domain actions mid-flight.
- When Microsoft provisioning is the backlog, scale the healthier satellite worker first. Add swap before adding another PowerShell lane, then add only one lane and watch memory/load before widening again.
- Do not borrow random active DigitalOcean servers just because they have spare RAM. For production migrations, either verify the server role/env and isolate the worker process, or create a purpose-built temporary worker box so fulfillment load cannot collide with unrelated services.
- Keep an upload-satellite path ready. An idle Pixie worker with Node, the backend bundle, Supabase env, and the Playwright browser cache can run `ReuploadWorker` lanes without touching the API web process.
- For same-day 100+ domain Instantly migrations, use a purpose-built upload-only host instead of pushing the API box past its CPU ceiling. An 8 vCPU / 16 GB temporary worker can run isolated `ReuploadWorker` lanes while the API host stays available for web/API traffic.
- On an 8 vCPU / 16 GB upload-only host, 20 Instantly upload lanes were stable with no swap and enough memory headroom; a cautious bump to 24 lanes was possible only after logs stayed clean. Treat that as a monitored ceiling, not a default.
- Pre-warm upload-only hosts completely before letting them claim actions: copy the backend bundle/env, run a Node require smoke test, install the Playwright browser, run `npx playwright install-deps chromium`, then launch a one-page Chromium test. Skipping `install-deps` caused immediate `libatk-1.0.so.0` browser-launch failures and noisy retry churn.
- If a new upload host fails during environment setup, stop its workers before fixing dependencies. Letting broken workers keep polling can churn through many pending domains, even if the actions remain retryable.
- For temporary DigitalOcean workers, create with an SSH key whose private half is actually present in the running Codex environment. The `codex-do-root-othma-msi` key was available in DO but not locally, creating an inaccessible temp droplet.
- Confirm the DO token can delete before relying on it for cleanup. In this run, the token could create droplets but returned 403 on delete, so temp worker IDs/IPs must be recorded for manual cleanup.
- Keep a Microsoft worker image promotion path ready too. Committing the patched container, copying its env, and loading it onto an idle Pixie worker let us add provision-only PowerShell lanes without rebuilding the worker from scratch.
- A non-container Microsoft runner is viable when the host already has PowerShell and `ExchangeOnlineManagement`; copy the patched worker into an isolated folder, add swap, set `WORKER_ACTION_TYPES=provision_inbox`, and start only a few lanes. First launch may spend ~15-20 seconds installing Node dependencies, so pre-warm temp workers before the bulk window.
- Settings validation should be a first-class finalization step: `daily_limit`, `sending_gap`, `enable_slow_ramp`, warmup enabled, warmup daily limit, and warmup reply rate should be read back from Instantly before marking the domain done.
- Treat Instantly warmup reply rate as a human percent in Simple Inboxes settings (`60` means `60%`). This run exposed a dangerous legacy decimal value (`0.6`) that Instantly displayed as `0.6%`; frontend labels/placeholders, backend storage normalization, worker payload normalization, and audits must all use/read back `60`.
- Instantly's accounts list/search endpoint already returns the settings fields needed for bulk proof (`daily_limit`, `sending_gap`, `warmup_status`, `warmup.limit`, and `warmup.reply_rate`). Use paged domain-level list/search validation for final audits instead of 1-by-1 account detail reads across tens of thousands of inboxes.
- Instantly tag assignment can accept a 100-account `toggle-resource` payload without applying every resource immediately. Final proof should read `custom-tag-mappings` with `tag_ids` plus `resource_ids=<email>` for only unresolved accounts; a broad tag mapping list can undercount at 10k+ mappings and should not be the only source of truth.
- If an upload retry fails after a transient settings timeout, reconcile before redoing OAuth. When the list/search endpoint already shows 99 accounts and 99 requested settings OK, it is safe to mark the reupload/provision actions complete from that direct audit instead of making the worker redo browser OAuth.
- Settings application cannot stay strictly one mailbox at a time for 99-inbox Azure orders. Keep separate throttles for settings PATCH and settings verification; a small concurrency is needed after OAuth, with 429/backoff handling, or the finalization phase becomes the bottleneck.
- Instantly may keep `enable_slow_ramp=true` on older accounts even after `warmup.increment` is set to `disabled`. Treat the warmup increment as the effective slow-ramp signal, and validate the user-requested fields directly: daily limit, sending gap, warmup status, warmup limit, and reply rate.
- If an Azure domain is recovered after a partial mailbox creation failure, reset/unblock all existing mailbox user passwords before upload, not only the newly created missing inboxes. Otherwise Instantly OAuth can hard-stop with "wrong password" even though Supabase has active inbox rows.
- For a domain-level wrong-password OAuth hard-stop, repair the whole domain, not just the first failed address. `prospectconversion.info` had multiple failing users after a recovery path; resetting/unblocking all 99 users from the stored Simple Inboxes passwords and requeuing the upload let the patched worker resume.
- Refresh Microsoft Graph tokens before late-stage password sync. A 99-inbox domain can spend long enough in mailbox checks, SMTP/IMAP confirmation, and DKIM that the Graph token acquired at worker start expires before `Sync-ActiveInboxPasswordsForUpload`; this showed up as 401s for every user on `conversionleverage.info`.
- Refresh Microsoft Graph tokens immediately before the final active-inbox password sync, not just at worker start. In this run, `growthpresence.info`, `clientpresence.info`, and `growthcredibility.info` showed the same late-stage password-sync failure class after long DKIM/readiness waits.
- Avoid `return $true` / `return $false` inside PowerShell `Where-Object` filters in shared queue helpers. On the new Microsoft satellite, this made `Get-PendingActions` return zero rows even though Supabase had pending work; use a local boolean expression and emit that boolean as the final statement instead.
- If mailbox creation returns 0 created inboxes, fail the provision action as well as the domain status. `acquisitionadvantage.info` hit a 0/99 mailbox-create result that only marked the domain `Both - Failed`, leaving the action `in_progress` with no live admin lock.
- A retry from `Both - Creating Mailboxes` must re-check Exchange accepted-domain state before creating mailboxes. `acquisitionadvantage.info` was manually requeued at the mailbox step, but Exchange no longer listed the domain as accepted, so all 99 mailbox creates failed with "not an accepted domain." The worker now defers back to `Both - DNS Records Added` when the accepted-domain proof is missing.
- Some Microsoft tenants can list the new domain as an authoritative accepted domain but still reject `New-Mailbox -PrimarySmtpAddress user@domain`. `acquisitionadvantage.info` showed this with an org-level `DefaultAcceptedDomain=null`. Do not switch the general Azure worker to `-MicrosoftOnlineServicesID`; Exchange reports parameter-set errors for shared-mailbox creates in this environment. Keep the proven general path and use a targeted tenant/domain recovery path for this class.
- The targeted Azure fallback for that class should only run after the normal shared-mailbox create fails with the accepted-domain error: create the mailbox with `New-Mailbox -MicrosoftOnlineServicesID user@domain` without `-Shared`, then convert it to shared with `Set-Mailbox -Type Shared` and normalize SMTP/Windows email fields. This avoids breaking the common path that is successfully creating most domains.
- If a domain already has 99 active inboxes and DKIM is tenant-enabled, do not rerun mailbox creation just to recover from password-sync token expiry. Reset/unblock the 99 users with a fresh Graph token, verify sample mailbox credentials, then queue `reupload_inboxes` with the original provision action id so upload finalization can complete the domain.
- Do not validate repaired Microsoft user passwords with an arbitrary public-client Graph `User.Read` scope. Some tenants reject that with first-party preauthorization errors even when the password is correct. Use the same resource-token pattern the worker/admin flow uses, or a real browser OAuth check, before deciding the credential is still bad.
- Microsoft OAuth can show an account-type fork when an address collides with a personal Microsoft account. The uploader should explicitly choose "Work or school account" before waiting for password, or it will time out on an otherwise valid mailbox.
- For bulk OAuth uploads, log per-domain upload start, per-25-account progress, settings apply start, settings verify result, and final uploaded/failed counts. Current logs are too sparse for ops triage.
- Keep action types filterable at runtime. During this migration, provisioning-only Microsoft workers prevented deletion/update workers from accidentally mixing with the one-off batch.
- Tenant-to-tenant Azure replacement needs a first-class batch launcher, not manual terminals. The launcher should consume a migration plan, require destination admins to pass message-trace validation, cap concurrency, and avoid running two domains from the same source tenant in one wave while still relying on admin row locks.
- Every single-domain fallback script must use the same full destination-validation report as the batch launcher. A stale top-10 validation default rejected clean idle tenants and stranded one resumed domain after cancellation; defaults must point at the canonical validation artifact, and clean-idle status (`auth_ok_no_recent_external_outbound`) must be accepted.
- Provision workers must self-quarantine premature migration actions. If a Jack threshold-migration `provision_inbox` action is picked up while the domain is still assigned to a thresholded source tenant, the worker should archive that action and log "move first" instead of retrying every poll and burning worker capacity.
- Credential/password-sync failures after mailbox creation should be controller-repairable but finite. If the domain has 99 active inbox rows on a clean destination admin, reset the action to rerun mailbox readiness from `Microsoft - Mailboxes Created`, while preserving the retry budget so persistent auth issues still surface.
- The batch launcher needs a closed-loop controller above it. The controller should repeatedly read canonical proof counts, open provision/upload backlog, and active launcher state; then either hold or launch the next safe wave. Every decision should be logged as JSONL with counts and the reason so ops can tune concurrency from evidence instead of manually watching terminals.
- Self-learning bulk policy should be feedback based: widen when fully proven/upload counts advance and open backlog is below ceilings; hold when DKIM/upload backlogs are at ceiling; never launch a duplicate domain while cancellation/provision actions are already open.
- DKIM CNAME upsert must be deployed into the actual running container filesystem and the image tag. The host had safe upsert code, but live Microsoft lanes still had an older script that treated duplicate CNAMEs as success and left selectors pointing at the old tenant.
- A DKIM retry loop should log the current and desired selector target when DNS verification fails. If the target is stale, the controller should repair/requeue immediately; waiting on Microsoft propagation cannot fix a wrong Cloudflare value.
- Pending Jack `microsoft_cancel_domain` actions caused by transient Exchange connection failures should be reusable by the migration wrapper. The batch launcher should continue to skip in-progress actions, but a single pending Jack cancellation action is safe to reclaim and retry rather than stranding the domain outside future waves.
- The controller needs self-repair hooks for known safe failure classes. In this run it now requeues only replacement upload/provision actions stranded by the known Instantly tag-mapping visibility lag, and logs the exact action ids it touched.
- Add a moved-domain provision-gap repair. If source teardown completes and the domain is already assigned to the clean destination admin but no replacement `provision_inbox` exists, the controller should detect it and queue the normal Azure worker instead of leaving the domain stranded.
- When repairing a moved-domain gap, learn the canonical inbox set from other domains in the same plan. `growthdevelopment.info` had 100 pending rows after teardown; the repair compared it to the canonical 99-name set, marked only the provable extra `laura@...` row deleted, and then queued the normal provision action.
- Proof readers must be resilient too. Supabase reads can time out during large runs; status/audit scripts should use smaller domain chunks plus retry/backoff so a transient API timeout does not stop the controller or get mistaken for a workflow failure.
- PowerShell worker helpers must suppress progress-save return values when assigning step objects. Letting `Persist-Progress` emit a boolean polluted `$step` variables and broke the cancellation worker before tenant teardown.
- Lease-fence timestamps must be normalized to invariant UTC strings before Supabase filters are built. Supabase returned ISO timestamps with offsets; PowerShell converted them to local display strings, which made safe fenced updates miss their own claimed action.
- Cancellation/replacement scripts need a source-tenant exception for thresholded admins. Creation workers must refuse `threshold exceeded` tenants, but teardown workers must be able to lock the assigned thresholded tenant to remove the old accepted domain safely.
- For live replacement scaling, prove one full domain end to end before widening the batch. After proof, run capped waves across unique source/destination admins; increase only after the cancellation, provisioning, upload, settings, and tag proof gates all pass.
- When widening, scale workers and controller ceilings together. Moving from 10 to 14 Microsoft provision-only lanes also required increasing the controller provision-open ceiling from 40 to 70; otherwise the new lanes could sit idle while the controller stayed artificially cautious.
- Upload workers can scale separately from Microsoft lanes. On the 8 vCPU / 16 GB upload host, raising from 4 to 8 low-concurrency `ReuploadWorker` lanes was safe because each worker kept Microsoft OAuth concurrency at 1 and settings/readback concurrency low.
- Cancellation must remove Microsoft recipients by stable exact identifiers, preferably primary SMTP/UPN, not Exchange display-name `Identity`. Reused persona display names caused ambiguous matches during tenant teardown; exact-address deletion plus Graph user cleanup let cancellation complete safely.
- Bulk proof scripts must page/chunk below the API row cap. A 75-domain inbox query silently undercounted 99-inbox domains because the REST response capped rows; use small chunks, roughly 8 domains at a time, so every domain can contribute all 99 inbox rows to the audit.
- Treat old upload actions as historical only. A replacement domain is not proven uploaded unless the completed `reupload_inboxes` action points to the new replacement `provision_inbox` action id and reports uploaded >= expected, failed = 0, and strict validation true.
- Destination pool validation should keep going after an admin auth/MFA failure and mark that admin unusable. A single destination admin requiring MFA must not stop the whole pool, but it also must not be selected for migration.
- Mailbox identity normalization must pass `-Confirm:$false` on `Set-Mailbox` calls that set `MicrosoftOnlineServicesID` / `WindowsEmailAddress`. Otherwise a non-headless worker can emit per-mailbox confirmation prompts and slow/noise up 99-inbox runs.
- Hot patches must update the runtime artifact the server actually imports. The upload host was running `dist/backend/src/workers/...`, so patching only TypeScript or stale `src` JavaScript would not change live behavior; every emergency worker patch needs a require/import smoke test against the exact entrypoint.
- Instantly tag mapping readback is eventually consistent. The upload worker can continue after a narrow "assignment accepted but mapping not visible yet" error only if final proof still requires tag mappings to be visible for every account before completion.
- Restarting upload workers can strand healthy in-progress upload actions until their lease ages out. A safe recycle path should explicitly requeue only replacement-linked in-progress uploads after an intentional worker restart.
- Failed-action dashboards must separate current replacement blockers from historical failures. Old failed cancellation/upload actions for domains that now have linked replacement upload proof should be hidden by default and available only with an historical flag.
- The controller should write a learning log separate from the decision log. Decisions explain why it held or launched; learnings classify current blocker patterns and DB final-proof counts so the next run can promote recurring one-off repairs into real gates.
- Aggressive backfill is safe only when driven by measured open slots. The controller can widen migration waves when provision/upload backlog is below ceilings and a live batch is not running, but it should still cap by open provision slots, upload pressure, and unique admin locks.
- Supabase API reads and idempotent writes need shared retry/backoff. Under a 45-domain local migration wave, individual source-teardown lanes hit empty Supabase errors while deleting inbox rows or writing action logs; the shared API helper now retries GET/PATCH/DELETE and action-log POSTs without retrying action-creating POSTs that could duplicate work.
- Scale upload and Microsoft lanes independently. In this run, upload was stable enough to increase from 8 to 12 upload workers while Microsoft was the bottleneck; Microsoft was widened from 14 to 17 lanes only after memory stayed healthy and swap remained unused.
- Action terminal states must match the database constraint. The correct safe terminal state for a premature replacement provision action is `cancelled`, not `archived`; use metadata and action logs to explain that only the bad action was cancelled, not the domain.
- The controller must own known safe repair loops, not depend on a human terminal. It now runs tag-lag requeue, moved-domain provision-gap repair, premature threshold-provision cancellation, DKIM-pending requeue, and credential-sync retry before every status decision.
- Do not start warmup until tenant-side DKIM is actually enabled. A domain can have correct Cloudflare CNAMEs that resolve publicly while Microsoft still rejects `Set-DkimSigningConfig`; hold upload/warmup and requeue DKIM enable rather than accepting DNS-only proof.
- Compute scaling has to respect the real bottleneck. At 17 remote Microsoft lanes plus 3 local provision lanes, both Microsoft and local hosts were memory-bound while upload workers had spare RAM; adding more provision lanes would increase failures, not throughput.
- Provision backlog is a deliberate throttle. When open provision work hits the ceiling, the controller should hold new tenant moves even if source domains remain, because moving faster than DKIM/mailbox/upload completion only increases queue depth and hides proof failures.
- Instantly cross-workspace account conflicts need mailbox-level recovery, not a full-domain retry. If one address already exists in another Instantly workspace, enqueue a Microsoft-backed `microsoft_update_inboxes` action for only that mailbox, remove the old address as an alias, then reset the linked replacement upload action. The other 98 mailboxes should not redo OAuth.
- The controller now has a reusable cross-workspace repair hook. It classifies `another workspace` errors, chooses unused replacement personas, queues the Microsoft mailbox mutation only after confirming 99 active rows on an Active destination admin, and leaves upload finalization blocked until the linked reupload action proves the repaired mailbox in Instantly.
- Final completion must run the expensive provider proof, not just DB proof. DB proof is useful for steering waves, but the controller must hold at the end until Instantly readback confirms count, settings, warmup reply rate, and tag mapping for every mailbox.
- Final provider proof for large migrations should run as partitioned manifests with bounded parallelism, not one giant serial sweep. Chunked proof keeps the process read-only, makes partial progress observable, and lets ops rerun only the failed chunk instead of re-reading every domain.
- Destination plan rows need an automatic clean-alternate repair path. Two final domains were blocked because their originally planned destination admins failed non-destructive validation (`Get-MessageTrace` error/MFA). The safe rule is: never force an error destination; instead choose an already validated-clean, Active, unlocked destination with a proven completed migration, write a plan backup, update the plan, and let the normal move-first controller continue.
- Completed work can still leave a stale domain UI state after retries. If a domain has the expected active inbox count, current admin equals the destination, replacement provision completed, and the linked replacement reupload completed, a controller repair can safely mark the domain `active / Both - Provisioning Complete`; final provider proof still remains the true completion gate.
- Status-mismatch proof reads must be batched or pre-filtered. A conservative per-domain proof repair worked but was slow under Supabase timeouts; the production worker should first filter only non-complete domain rows, then batch action/inbox/admin evidence.
- Pending actions can be stuck without appearing as failed. `conversionscalability.info` exhausted its `Exchange sync pending after 600 seconds` retry budget and stayed `pending`, which normal workers skip because attempts reached max. The controller needs a class-specific requeue that only extends the budget before mailbox creation, while the domain is on an Active destination, has 99 pending inbox rows, and has zero active inbox rows.
- Extra compute must run filtered workers, not the broad Microsoft orchestrator. Starting generic local `run.ps1` loops can pick unrelated Microsoft recovery work; scale this migration with provision-only / Jack-scoped lanes, or immediately requeue any interrupted Jack action with the retry budget preserved.
- The generic worker needs first-class source filters, not just action-type filters. `run.ps1` now supports `WORKER_ACTION_SOURCE` / `WORKER_ACTION_SOURCES` so emergency capacity can run only `jack_threshold_tenant_migration` provision actions while leaving unrelated recovery/cancellation queues alone.
- Remaining confirmation prompts came from a few Set-Mailbox/Set-User calls missing `-Confirm:$false`. The live mounted Microsoft worker script was patched so future action starts suppress conversion/name-update prompts; this should be part of the committed worker image, not just a host hot patch.

## App and Admin Workflow Updates

- Customer-facing domain counts must come from canonical `inboxes` rows for the domain, not from the original order config, seeded payload, or a cached `config.inboxes` object. During this run, the UI showed `1 inbox`, `92 inboxes`, or `0 inboxes` for domains that direct database and Instantly audits proved had 99 active inboxes.
- The orders API must paginate inbox lookups for large batches. A single Supabase `.in(domain_id, ids)` read can silently hit the default row cap across 100+ domains, which makes later domains appear to have `1`, `0`, or partial inbox counts even when the canonical rows exist.
- Domain detail modals should use the same canonical source as the list view. `revenuecraft.info` showed an empty inbox modal even though the domain had 99 active database rows and 99 Instantly accounts with the requested settings.
- Provider badges and per-inbox launch labels must be derived from the domain/provider record and credential type, not stale per-inbox/order fallback data. Azure domains were visually mixed with Gmail labels in the modal, which creates a false support signal.
- Completed status must require a final proof record, not only a UI status string. For Azure this should mean: domain active, provider action completed, sending-tool upload action completed, expected active inbox count present, and direct sending-tool validation proving expected account count and requested settings.
- Treat post-upload "repairs" as product gaps, not normal ops. Each one-off reconciliation from this run must become a named worker/audit gate: settings unit normalization, tag assignment proof, account-count proof, provider mailbox-count proof, and UI count consistency. Once a gate exists, the domain cannot go green until the gate passes.
- Rename operator-facing "repair" flows that are really final proof/reconciliation gates. Ops should see "settings audit", "tag proof", or "provider count proof", not a stream of repair jobs that makes a healthy bulk run look broken.
- Add live upload telemetry per domain: target count, provider-visible count, OAuth completed count, settings-ok count, tag-ok count, current mailbox email, last browser checkpoint, and last provider API read. During this run, browser processes and Instantly account samples proved forward motion even when logs were sparse.
- Final proof scripts must return a failing exit status when provider evidence is incomplete. Printing `missing`, `badAfter`, or `untaggedAfter` counts is not enough for production automation.
- Bulk ops repair and proof entrypoints should accept manifest/file inputs instead of shell-escaped domain arrays. The fragile path in this run was ad hoc `pwsh -Command` text and manually quoted arrays, not the file-based worker path that actually processed the domains.
- Add a reconciler for impossible final states: if provider and upload actions are completed but the domain row remains `in_progress`, re-read canonical evidence before repair. The required proof should be active inbox count, sending-tool account count, settings readback, tag mapping, and completed action results.
- If an upload action exhausts retries only because tag mappings were not visible yet, do not redo OAuth. Re-read the account list, tag mappings, and settings; patch only the missing settings/tag state, then finalize from direct provider proof.
- Store and display audit fields for admins: expected inbox count, active Simple Inboxes row count, sending-tool account count, settings-ok count, last audit time, and the audit source. This lets ops answer "is it really complete?" without re-running ad-hoc scripts.
- For bulk replacement orders, consider hiding or locking the customer-facing per-inbox cards until final validation is complete. Show a domain-level ready state and admin-only detail instead of exposing partial/stale inbox detail to the customer.
- Normal Azure orders use 25 inboxes, but they need the same completion gates and UI data path as this 99-inbox stress run. The expected count should be order/domain-specific, not hard-coded to either 25 or 99.
- Admin bulk views need mismatch filters: expected vs active rows, active rows vs sending-tool accounts, settings mismatch, provider complete but upload pending, and UI-visible count mismatch. These are the operational views we kept rebuilding manually during this migration.
- If a replacement migration seeds domains before provisioning finishes, show the customer a true intermediate state such as "processing domain" or "uploading inboxes", but never show a fabricated one-inbox default.
- Export CSV should only use canonical active inbox rows after final provider/upload proof. It should not export from the original requested inbox list when provider creation or sending-tool upload is still in progress.

## Risks Found

- The Microsoft action lease heartbeat returned `401` until the worker started sending the expected API user agent. Heartbeat failures are high risk for 99-inbox domains because stale actions can be reclaimed mid-run.
- The current launcher is one worker loop by default. Bulk Azure orders need multiple loops or a first-class worker pool with memory guardrails.
- DNS activation can take longer than the first 15-minute wait; the process should not treat that as a hard failure when the registrar update succeeded.
- The live API server had drifted from the local worker code. Production deploys need a single release artifact or a deploy check that confirms the running `dist` code includes required worker lease/OAuth/settings behavior.
- Instantly account objects may omit requested fields until settings are applied/read back; do not infer success from account existence alone.
